# Mixtape Bug Hunt — Submission

## AI Usage

I used Claude (Sonnet 5, via Claude Code) throughout this project as a pairing partner for navigation and verification, not as a source of unverified diagnoses. Specific uses:

- **Codebase orientation**: Had the assistant read `app.py`, `models.py`, every file in `routes/` and `services/`, and `seed_data.py` in full before touching any bug, to build the codebase map below. This was plain reading/summarizing, not guessing.
- **Tracing call chains**: For each issue, traced the path route → service function myself with the assistant reading the code aloud (e.g. `POST /songs/<id>/rate` → `routes/songs.py:rate()` → `notification_service.rate_song()`), rather than jumping straight to the service file.
- **Hypothesis verification, not hypothesis generation**: For every bug, I (via the assistant) read the suspicious function first, formed a hypothesis about the root cause, and then verified it by *running* the code — a Python one-liner script importing `create_app`/`db`/the service function and calling it with controlled inputs, plus `curl` against the running dev server. I did not ask the assistant to "find the bug" blind; each RCA below is backed by an actual command whose output is in my history.
- **Where AI helped understand something non-obvious**: For Issue #3, my first assumption (an `outerjoin` to `song_tags` causing row fanout) tested out *incorrectly* when I ran the existing pytest suite — all tests passed even before any fix. I asked the assistant to help figure out why, and by directly comparing `db.session.query(Song)...all()` against a raw `select(Song)...execute().scalars().all()` in a Python shell, we discovered that this installed SQLAlchemy version's legacy `Query.all()` auto-uniques full-entity results, masking the fanout — while the equivalent modern `select()`-style call (and raw SQL) still returns 3 duplicate rows for a 3-tag song. This is a case where reading the code alone wasn't enough; execution tracing (not AI guessing) resolved the ambiguity. The fix (removing the unnecessary join) addresses the real defect regardless of which query style masks it.
- **Where I overrode/verified rather than trusted**: For Issue #2, the "24 hour" window read as plausible "recent" logic on its own. I didn't take a first impression at face value — I reproduced the reported symptom directly (queried a friend whose only listening event was ~2 hours old and confirmed it appeared in "Listening Now"), then used the seed data's own comments (`"Recent events (within the past 30 minutes) — should appear"`) as the spec for the correct threshold, rather than picking an arbitrary number.

No code was generated by AI without me first reading the surrounding function and confirming the diagnosis by running it.

## Codebase Map

### Main files

- **`app.py`** — Flask application factory (`create_app`). Initializes `SQLAlchemy`, registers the four blueprints (`songs`, `playlists`, `users`, `feed`) under their URL prefixes, and calls `db.create_all()`. Must be started with `FLASK_APP=app:create_app flask run` (not `python app.py`) to avoid a double-import of the `db` object.
- **`models.py`** — All SQLAlchemy models: `User`, `Tag`, `Song`, `ListeningEvent`, `Rating`, `Playlist`, `Notification`, plus three association tables: `friendships` (symmetric many-to-many between users), `song_tags` (many-to-many between songs and tags), and `playlist_entries` (many-to-many between playlists and songs, extended with `position`, `added_by`, and `added_at` columns — songs in a playlist have an explicit order, not just insertion order).
- **`routes/`** — Thin Flask blueprints. Every route parses the request, calls exactly one service function, and formats the JSON response / catches `ValueError` into a 4xx. No business logic lives here.
  - `routes/songs.py` — search, song detail, rating, listen events.
  - `routes/playlists.py` — playlist creation, detail, song listing, adding songs.
  - `routes/users.py` — user detail, streak, notifications.
  - `routes/feed.py` — "listening now" and general activity feed.
- **`services/`** — All business logic. This is where all five bugs live.
  - `streak_service.py` — computes/updates a user's consecutive-day listening streak.
  - `feed_service.py` — "Friends Listening Now" (recency-filtered) and the general activity feed (unfiltered, limit-based).
  - `search_service.py` — song search by title/artist.
  - `notification_service.py` — creates and retrieves `Notification` rows; two producers exist (playlist-add, rating) and two consumers (list, mark-read).
  - `playlist_service.py` — playlist creation and ordered song retrieval.
- **`seed_data.py`** — Populates the DB with 5 users, 13 songs (with 0/1/3+ tags to specifically expose the search duplicate bug), 3 playlists, friendships, listening events at controlled ages (both inside and outside the "recent" window), and one working playlist-add notification (deliberately, so the correct pattern is visible for comparison against the broken rating path).
- **`tests/`** — `test_streaks.py`, `test_search.py`, `test_playlists.py` shipped with the starter and already encode the expected (fixed) behavior for issues #1, #3, #5. `test_notifications.py` is the regression test I added for issue #4.

### Data flow — a friend rates your song (Issue #4's territory)

1. Client sends `POST /songs/<song_id>/rate` with `{"user_id", "score"}`.
2. `routes/songs.py:rate()` parses the body and calls `notification_service.rate_song(user_id, song_id, score)`.
3. `rate_song()` validates the score (1–5), loads the `Song` and the rating `User`, then either updates an existing `Rating` row (enforced unique by a `UniqueConstraint("user_id", "song_id")`) or inserts a new one, and commits.
4. **(after my fix)** if the rater isn't the song's original sharer, `rate_song()` calls `create_notification()` to insert a `Notification` row for `song.shared_by`, with `notification_type="song_rated"`.
5. The sharer later hits `GET /users/<user_id>/notifications`, which calls `notification_service.get_notifications()`, returning all `Notification` rows for that user ordered by `created_at` descending.

The equivalent flow for adding a song to a playlist (`POST /playlists/<id>/songs` → `routes/playlists.py:add_song()` → `notification_service.add_to_playlist()`) already had this last step; rating didn't, until fixed.

### Pattern noticed

Every route delegates immediately to a service function and does no business logic of its own — routes only parse input and shape the response, and raise/catch `ValueError` for 404/400s. All actual logic, including notification side effects, lives in `services/`. The two "producer" functions in `notification_service.py` (`add_to_playlist`, `rate_song`) are supposed to follow the same shape (look up the entities → do the primary write → conditionally notify the sharer), which made the missing step in `rate_song()` easy to spot once `add_to_playlist()` was read first as the reference implementation.

## Root Cause Analysis

### Issue #1 — My listening streak keeps resetting

**How I reproduced it**: Ran `update_listening_streak()` directly in a Python shell with two controlled `datetime` values — a Saturday and the following Sunday (both timezone-aware UTC) — simulating a user who listens on consecutive days. This is also exactly what the shipped `tests/test_streaks.py::test_streak_increments_on_sunday` does; running `pytest tests/test_streaks.py` before any fix showed this one test failing (`assert 1 == 2`) while the other four streak tests passed.

**How I found the root cause**: Started at `routes/songs.py:listen()` → `streak_service.record_listening_event()` → `update_listening_streak()`. The function is short (services/streak_service.py:42-78) and the day-boundary logic is isolated to one `if/elif/else` block, so no further tracing was needed — reading the `elif` condition against Python's `datetime.weekday()` semantics (Monday=0 … Sunday=6) was enough to confirm the defect.

**The root cause**: `elif days_since_last == 1 and today.weekday() != 6:` only incremented the streak on a consecutive day if today was *not* Sunday. Since `weekday()` returns `6` for Sunday, that extra clause evaluates to `False` every Sunday, so the branch falls through to `else: user.listening_streak = 1` — resetting a perfectly valid, still-consecutive streak to 1 once a week, every week, on Sunday.

**My fix and side-effect check**: Removed the `and today.weekday() != 6` clause entirely, leaving `elif days_since_last == 1:`. There was no legitimate reason in the spec (docstring: "If the user listened yesterday: streak increments by 1") for Sunday to be a special case. Verified all 5 tests in `test_streaks.py` pass, including same-day (no double count), skipped-day (resets to 1), and the Sunday boundary in both directions (Sat→Sun increments; a genuine skipped day still resets even when landing on Sunday, since `days_since_last` would be `>1` in that case, untouched by this fix).

### Issue #2 — Friends Listening Now shows people from yesterday

**How I reproduced it**: Called `feed_service.get_friends_listening_now(kenji_id)` directly against the seeded database, where `kenji`'s only friend with any listening history other than a same-day one was `nova`, whose most recent event was recorded ~2 hours earlier — at `22:42` the previous calendar day, since the request was made shortly after midnight. The buggy code returned `nova` in the "listening now" feed despite that listen being from "yesterday" by any calendar-day definition of the feature's name.

**How I found the root cause**: `routes/feed.py:listening_now()` → `feed_service.get_friends_listening_now()`. The function builds `cutoff = datetime.now(timezone.utc) - RECENT_THRESHOLD` and filters `ListeningEvent.listened_at >= cutoff`, which is logically correct filtering — the problem wasn't the comparison, it was the constant: `RECENT_THRESHOLD = timedelta(hours=24)` (services/feed_service.py:13). A 24-hour rolling window is far too generous for a feature literally named "Listening Now" (implying real-time presence), and `seed_data.py`'s own comments confirm the intended distinction: events "within the past 30 minutes" are the ones meant to appear, while everything older (including the 2–18 hour range used for other seeded events) is meant to be excluded.

**The root cause**: `RECENT_THRESHOLD` was set to 24 hours instead of a real-time window, so any friend who listened at any point in the last full day — including well into "yesterday" from the viewer's current moment — counted as "currently listening."

**My fix and side-effect check**: Changed `RECENT_THRESHOLD` to `timedelta(minutes=30)`. Verified with fresh seed data that (a) `kenji` viewing `nova` (2-hour-old event) now correctly returns an empty feed, and (b) `nova` viewing `darius`/`simone`/`kenji` (each with genuine 10–20-minute-old events) still returns all three — confirming the fix narrows the window without breaking the "actually recent" case. Also checked `get_activity_feed()` (the general activity feed, which is intentionally unfiltered by recency per its own docstring) was untouched, since it doesn't use `RECENT_THRESHOLD` at all.

### Issue #3 — The same song keeps showing up twice in search

**How I reproduced it**: The shipped `tests/test_search.py` already encodes the expected bug (`test_search_no_duplicates_multi_tag_song`, with a comment "Should be 1, bug causes it to be 3"). Running `pytest tests/test_search.py` before any fix, however, showed all 5 tests passing — including that one. Investigating further, I called `search_songs("Crown Heights")` (a song seeded with 3 tags) directly and got exactly 1 result both via the ORM and via `pytest`. I then dropped to raw SQL / SQLAlchemy 2.0-style `select()` execution to check what the *query itself* returns before any ORM-level deduplication: `select(Song).outerjoin(song_tags, ...).filter(...)` executed via `session.execute(stmt).scalars().all()` (no `.unique()`) returned **3** rows for that one song — one per tag row the join fanned out into.

**How I found the root cause**: `routes/songs.py:search()` → `search_service.search_songs()`. The function performs `db.session.query(Song).outerjoin(song_tags, Song.id == song_tags.c.song_id).filter(...)`, but never uses any column from `song_tags` in the `WHERE` clause or `SELECT` — tags are loaded separately per-song via the `Song.tags` relationship inside `to_dict()`. The join exists for no functional reason. Confirming this was the actual defect (not just a hunch) required comparing the legacy `db.session.query(...).all()` path against the modern `select(...).execute().scalars().all()` path side-by-side, since the installed SQLAlchemy 2.0.51's legacy `Query.all()` turned out to auto-unique full-entity result sets — silently masking the row fanout in this environment, even though the underlying SQL is still wrong.

**The root cause**: The `outerjoin` to `song_tags` in `search_songs()` produces one result row per matching tag association (a join table row per tag), so a song with N tags contributes N rows to the query result. This is a classic join-fanout bug. It happened to not manifest as visible duplicate dicts in this exact dependency version because legacy ORM `Query.all()` auto-deduplicates by primary key — but the query is fragile and produces genuine duplicate rows at the SQL level, which would surface as duplicates under `select()`-style execution, raw SQL, or a different SQLAlchemy/ORM configuration.

**My fix and side-effect check**: Removed the `.outerjoin(song_tags, ...)` call entirely (and the now-unused `Tag`/`song_tags` imports), since the join served no filtering or selection purpose. Verified: `pytest tests/test_search.py` still passes all 5 tests, and — more importantly — re-ran the raw `select()`/`execute().scalars().all()` check and confirmed it now returns exactly 1 row for the 3-tag song, proving the fix addresses the actual SQL-level defect rather than relying on the ORM's masking behavior.

### Issue #4 — I got notified when a friend added my song to a playlist but not when they rated it

**How I reproduced it**: Started the dev server, found a song shared by `nova`, then issued `POST /songs/<id>/rate` as `darius` (a different user). The request succeeded (201, rating saved), but a subsequent `GET /users/<nova_id>/notifications` showed only the one pre-existing `song_added_to_playlist` notification from seed data — no new notification was created for the rating.

**How I found the root cause**: `routes/songs.py:rate()` → `notification_service.rate_song()`. Compared this function line-by-line against `notification_service.add_to_playlist()`, which is the working reference pattern: `add_to_playlist()` performs its primary write (adding the song to the playlist), then checks `if song.shared_by != added_by_user_id:` and calls `create_notification(...)`. `rate_song()` performs its primary write (saving the `Rating`, `db.session.commit()`) and then just `return`s — there is no call to `create_notification()` anywhere in the function.

**The root cause**: The rating code path was never wired up to the notification system. It's not a typo or an off-by-one — the notify step simply doesn't exist in `rate_song()`, while the exact same architectural pattern (check `sharer != actor`, then `create_notification()`) is fully implemented one function away in `add_to_playlist()`.

**My fix and side-effect check**: Added, after the existing `db.session.commit()` in `rate_song()`, the same guarded `create_notification()` call used in `add_to_playlist()` — `notification_type="song_rated"`, skipped when `song.shared_by == user_id` (rating your own song shouldn't self-notify). Verified via direct calls: (a) a friend rating your song creates exactly one `song_rated` notification, (b) rating your own song creates zero notifications, and (c) updating an existing rating (same user, new score) still fires a fresh notification rather than silently no-op'ing. Added `tests/test_notifications.py` to lock this in (see Regression Test section).

### Issue #5 — The last song in a playlist never shows up

**How I reproduced it**: Called `playlist_service.get_playlist_songs()` against a seeded playlist known to contain 7 songs and got back 6. The shipped `tests/test_playlists.py::test_playlist_returns_all_songs` encodes this exactly (comment: "Bug causes this to return 4" for its 5-song fixture) — running `pytest tests/test_playlists.py` before any fix confirmed both `test_playlist_returns_all_songs` and `test_playlist_returns_songs_in_order` failing, with the last track (`"Track 5"`) missing from the ordered list.

**How I found the root cause**: `routes/playlists.py:get_songs()` → `playlist_service.get_playlist_songs()`. The function correctly joins `Song` to `playlist_entries`, filters by `playlist_id`, and orders by `position` ascending — that part is right. The very last line, though, is `return [song.to_dict() for song in songs[:-1]]` — a Python list slice that drops the final element of `songs` before converting to dicts, regardless of how many songs are in the playlist.

**The root cause**: An errant `[:-1]` slice on the already-correctly-ordered `songs` list silently truncates the last song from every playlist listing, no matter the playlist's length. The function's own docstring even asserts "This function returns all songs in the playlist" — directly contradicted by the code beneath it.

**My fix and side-effect check**: Removed the `[:-1]` slice, returning `[song.to_dict() for song in songs]`. Verified both boundary sides: a 7-song seeded playlist now returns 7 songs (was 6), and the existing `test_empty_playlist_returns_empty_list` test (0 songs → empty list, where `[:-1]` on `[]` is also `[]`, so this edge case was already accidentally correct) still passes — confirming the fix doesn't regress the empty-playlist case.

## Regression Test

Added `tests/test_notifications.py`, covering Issue #4 (no test file existed for the notification service before this project):

- `test_rating_a_friends_song_notifies_the_sharer` — a friend rating your song produces exactly one `song_rated` notification.
- `test_rating_your_own_song_does_not_self_notify` — rating your own song produces zero notifications.
- `test_updating_an_existing_rating_notifies_again` — changing an existing rating still fires a notification.

Verified these tests fail against the pre-fix `notification_service.py` (checked out from the commit immediately before `00eb5dc`) with `assert 0 == 2`-style failures, and pass against the fixed version — confirming they would have caught this bug before it shipped.
